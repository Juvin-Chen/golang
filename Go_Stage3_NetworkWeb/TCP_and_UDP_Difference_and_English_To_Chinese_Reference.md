### 二、UDP 代码术语与流程解析

#### 1. 客户端代码术语映射

```go
serverAddr, _ := net.ResolveUDPAddr("udp", "127.0.0.1:9999")
// parse -> 将字符串地址解析为 UDPAddr 对象
// scheme -> "udp" 表示协议
// host + port -> "127.0.0.1:9999" 指定目标地址

conn, err := net.DialUDP("udp", nil, serverAddr)
// scheme -> 协议类型
// addr -> serverAddr 指定目标地址
// err -> 错误处理（如端口占用）

data, err := json.Marshal(p)
// marshal -> 结构体转为 JSON 字节流

conn.Write(data)
// raw data -> 发送原始字节流
```

#### 2. 服务端代码术语映射

```go
addr, _ := net.ResolveUDPAddr("udp", ":9999")
// host -> 空 IP 表示监听所有网卡
// port -> 9999 监听端口

conn, err := net.ListenUDP("udp", addr)
// scheme -> 协议类型
// addr -> 绑定本地地址

n, clientAddr, err := conn.ReadFromUDP(buf)
// buffer -> 接收数据的缓冲区
// query -> clientAddr 包含客户端 IP+端口（类似查询来源）

err = json.Unmarshal(buf[:n], &p)
// unmarshal -> 解析 JSON 字节流到结构体
```

### 三、TCP 与 UDP 术语对比（结合握手🤝过程）

| 术语       | TCP                                       | UDP                                            | 说明                                   |
| :--------- | :---------------------------------------- | :--------------------------------------------- | :------------------------------------- |
| `Dial`     | `net.Dial("tcp", "ip:port")` 触发三次握手 | `net.DialUDP("udp", nil, addr)` 仅本地分配端口 | TCP 阻塞等待握手，UDP 无               |
| `Listen`   | `net.ListenTCP("tcp", addr)`              | `net.ListenUDP("udp", addr)`                   | TCP 面向连接，UDP 面向数据报           |
| `Accept`   | `listener.Accept()` 等待客户端连接        | 无                                             | TCP 需要 `Accept`，UDP 直接 `ReadFrom` |
| `err` 处理 | 握手失败时返回错误                        | 发送/接收时可能失败                            | TCP 更严格的错误检查                   |

### 四、常见问题术语解答（结合你的疑问）

#### 1. 为什么客户端和服务端都有 `ResolveUDPAddr`？

- 术语解释：

  - `parse`：解析地址字符串到 `UDPAddr` 对象。
  - `host` 和 `port`：客户端需指定具体目标（如 `127.0.0.1:9999`），服务端监听所有网卡（`:9999`）。

- 代码示例：

  ```go
  // 客户端：指定 host + port
  serverAddr, _ := net.ResolveUDPAddr("udp", "127.0.0.1:9999")
  
  // 服务端：仅指定 port（host 为空）
  addr, _ := net.ResolveUDPAddr("udp", ":9999")
  ```

#### 2. TCP 的三次握手去哪了？

- 术语解释：

  - `Dial` 函数内部自动完成握手。
  - `err` 用于捕获握手失败（如服务端未启动）。

- 代码验证：

  ```go
  conn, err := net.Dial("tcp", "127.0.0.1:9999")
  if err != nil {
      panic("三次握手失败: " + err.Error()) // 若服务端未运行，会触发 panic
  }
  ```

#### 3. 为什么 UDP 没有握手？

- 术语解释：

  - `DialUDP` 仅分配本地端口（无网络交互）。
  - `Write` 时直接发送数据包，不保证送达。

- 代码验证：

  ```go
  // 即使服务端未启动，DialUDP 也会成功
  conn, _ := net.DialUDP("udp", nil, serverAddr)
  conn.Write([]byte("可能丢失的数据")) // 若服务端未运行，数据包会丢失
  ```

## 🌐 UDP 无连接通信核心原理：为什么没有“握手”？

> **结论：UDP 服务端的 `ReadFromUDP` 是阻塞等待** 数据包**，不是等待连接（握手）！**

### 🔍 核心问题澄清

| 可能的疑问                     | 正确解释                                         | 为什么？                                                     |
| :----------------------------- | :----------------------------------------------- | :----------------------------------------------------------- |
| “`ReadFromUDP` 是阻塞握手吗？” | ❌ **不是** ✅ **是阻塞等待客户端发送 UDP 数据包** | UDP 是 **无连接协议**，不需要建立连接，所以没有握手过程。`ReadFromUDP` 会一直等待**数据包到达**（不是等待连接建立） |

------

### 📌 为什么没有“握手”？（UDP 本质）

| 协议特性       | TCP                             | UDP                                            | 说明                                     |
| :------------- | :------------------------------ | :--------------------------------------------- | :--------------------------------------- |
| **连接建立**   | ✅ 三次握手（`Dial`/`Accept`）   | ❌ **无连接**                                   | UDP 通信前**不需要**建立连接             |
| **通信方式**   | 面向连接                        | 面向数据报                                     | UDP 直接发送数据包，不关心对方是否在线   |
| **服务端启动** | `Listen` → `Accept`（等待连接） | `Listen` → `ReadFromUDP`（**直接等待数据包**） | UDP 服务端启动后**立刻进入等待数据**状态 |
| **阻塞原因**   | 等待连接（握手）                | 等待**数据包到达**                             | `ReadFromUDP` 阻塞 = 等数据，不是等连接  |

------

## 🧪 代码验证：UDP 服务端启动流程

```go
// 1. 监听端口（无连接，直接开始等待）
addr, _ := net.ResolveUDPAddr("udp", ":9999")
conn, _ := net.ListenUDP("udp", addr)

// 2. 立刻进入阻塞等待（不等待任何握手！）
for {
    n, clientAddr, err := conn.ReadFromUDP(buf) // ← 此处阻塞等待数据包
    // ... 处理数据
}
```

> **关键点**：服务端调用 `ListenUDP` 后，**立即**进入 `ReadFromUDP` 阻塞，**不等待**客户端连接请求（因为 UDP 没有连接！）

> **任何人**（客户端）把包裹（数据包）扔进邮箱，服务端立刻收到（`ReadFromUDP` 返回）。

## ❌ 常见误解（为什么有人会误以为有握手？）

| 误解                               | 事实                                | 原因                         |
| :--------------------------------- | :---------------------------------- | :--------------------------- |
| “`ReadFromUDP` 是阻塞，所以是握手” | ❌ 阻塞等待的是 **数据包**，不是连接 | 混淆了“阻塞等待”和“握手”     |
| “TCP 有握手，所以 UDP 也应该有”    | ❌ UDP 从设计上就**没有**握手        | UDP 为**低延迟**牺牲了可靠性 |

## 💡 为什么 UDP 设计成“无连接”？

| 优势             | 说明                               |
| :--------------- | :--------------------------------- |
| **零握手延迟**   | 数据可立即发送（无需等待 RTT）     |
| **低开销**       | 内核无需维护连接状态（TCB）        |
| **高吞吐**       | 单机支持百万级并发（TCP 难以企及） |
| **实时场景刚需** | 视频流、在线游戏等对延迟敏感的场景 |

> 📌 **典型场景**：直播推流（UDP） vs 文件下载（TCP）
> *直播需要秒级延迟，UDP 的“无握手”是核心优势*

## 📌 重要总结

>  ✅ `ReadFromUDP` 阻塞等待的是 **数据包**（客户端发送的数据），不是“握手”。

| :----------------------- | :---------------------------------------- |

| 为什么代码里看不到握手？ | ✅ 因为 **UDP 从设计上就不要握手**         |
