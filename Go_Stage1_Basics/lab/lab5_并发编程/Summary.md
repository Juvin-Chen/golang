### 📝 lab5 实验总结

#### 1. 核心模式（Design Patterns）

- **生产者-消费者模式：** 通过 Channel 解耦任务的生成与处理。生产者只管往 Channel 塞数据，消费者只管从 处理者的Channel 读数据，互不干扰。
- **Worker Pool (工人池) 模式：** 控制并发数量的终极方案。通过启动固定数量的 Goroutine（工人），竞争读取同一个 Channel（任务池），实现天然的负载均衡，防止瞬时高并发压垮服务器。

#### 2. 通道（Channel）的底层真相

- **关闭不是销毁：** `close(ch)` 只是在通道尾部挂上“停止接单”的标志，意味着不会再往通道里面传数据。通道里原有的数据不会消失，依然可以被正常读取。
- **优雅退出的关键：** `for data := range ch` 是最地道的读通道方式。它会持续阻塞等待，**直到通道被 `close()` 且里面没有剩余数据**，循环才会自动结束。
- **谁生产谁关闭：** 永远不要在接收方关闭通道，也不要在有多个并发发送方时由某一个发送方贸然关闭。

#### 3. 并发同步的“三件套”

- **`sync.WaitGroup`（计步器）：** 解决“主协程不等子协程”的问题。`Add()` 增加任务数，`Done()` 完成任务数，`Wait()` 原地阻塞直到计数归零。
- **关灯协程（后台监工）：** 当有多个 Goroutine 共同向一个 Channel 发送结果时，主协程不能直接 `close`。必须单独起一个 `go func()` 执行 `wg.Wait()`，确认所有人下班后，再执行 `close(results)`。
- **`context` 与 `select`（全厂广播与紧急制动）：**
  - `for range` 只能“死盯”一个通道。
  - `select` 可以“眼观六路”，同时监听多个通道（比如任务通道和取消信号通道）。
  - `context.WithCancel()` 提供了一个可以在 Goroutine 树之间瞬间传递的“红色紧急按钮”，一键取消所有关联的后台任务。

#### 4. 🚨 `main` 函数的执行顺序玄机（兵马未动，粮草先行）/ 可参考project2

- **避坑指南：** 绝对不要在 `main` 函数中“先往通道塞满任务，再去启动 Worker 协程”。因为 Go 的通道发送是阻塞的，一旦通道满了，主协程就会卡死在发送动作上，导致后续启动 Worker 的代码永远无法执行（触发死锁 Deadlock）。
- **正确姿势：先启动消费者（Worker），再启动生产者（发任务）。** 让工人们先在传送带末端就位并挂起等待，主协程再开始往通道里塞数据。这样塞一个就被抢一个，传送带永远不会堵死。
